What is API Testing?

API testing checks that an application's programming interface works correctly — sending requests, validating responses, and making sure data flows properly between systems. At the intermediate level, you go beyond "does it return 200?" to asking why, how fast, and what happens when things go wrong.

A REST API is like a waiter at a restaurant. You (the client) place an order (request), the waiter (API) passes it to the kitchen (server), and brings back your food (response).

Key Testing Types

Testing Checklist

  • Verify correct HTTP method is used
  • Validate response status code matches expectation
  • Check response body schema and data types
  • Confirm authentication tokens are handled correctly
  • Test with invalid inputs (negative testing)
  • Measure response time against SLA
  • Test edge cases (empty strings, null, huge numbers)

HTTP Methods Deep Dive

Each HTTP method has a specific purpose. A good API tester knows not just what they do, but what to verify when testing each one.

GETRetrieve data

Fetches a resource without modifying it. Should be safe and idempotent.

GET /api/users/123
✓ Test: Response body matches expected schema. Multiple identical GETs return same result. No side effects on server.
POSTCreate a resource

Sends data to create a new resource. Not idempotent — calling it twice creates two records.

POST /api/users Body: { "name": "Bob" }
✓ Test: Returns 201 Created. Location header contains new resource URL. Duplicate POST creates second record.
PUTReplace a resource

Replaces the entire resource. Idempotent — calling it multiple times has the same effect.

PUT /api/users/123 Body: { "name": "Bob", "email": "bob@test.com" }
⚠ Missing fields in PUT body are treated as null/removed. Test that omitting a field doesn't silently delete data.
PATCHPartially update

Updates only the fields you send. Other fields remain unchanged.

PATCH /api/users/123 Body: { "name": "Bobby" }
✓ Test: Only the patched field changed. All other fields remain exactly as before.
DELETERemove a resource

Deletes the specified resource. Should be idempotent — deleting twice shouldn't cause an error.

DELETE /api/users/123
✓ Test: Returns 204 No Content. Second DELETE returns 404. GET after DELETE returns 404.

Authentication & Security Testing

Security bugs in APIs are extremely common and often high-severity. As an intermediate tester, you should actively look for these.

Common Auth Methods

API Keys

A static string sent with every request. Simple but less secure than token-based approaches.

GET /api/data?api_key=abc123 // or in header: X-API-Key: abc123
⚠ Test: What happens if the key is expired? Shared between users? Sent over HTTP not HTTPS?

Bearer Tokens (JWT)

A signed token containing claims about the user. Most common in modern REST APIs. The server verifies the signature without checking a database.

Authorization: Bearer eyJhbGci...
⚠ Test: Expired token → 401? Tampered token → 401? Token from User A used for User B's data → 403?

OAuth 2.0

An authorization framework where users grant apps limited access to their account.

Authorization Code
Web apps (most secure)
Client Credentials
Machine-to-machine
Implicit
Legacy (avoid)
Device Code
Smart TVs, CLI tools

Security Test Cases to Always Run

  • Access another user's resource with your token (BOLA / IDOR)
  • Send a request with no auth header → expect 401
  • Send an expired token → expect 401
  • Modify JWT payload (change user ID) → expect 401
  • Try admin endpoints as a regular user → expect 403
  • SQL injection in query params: ?id=1' OR '1'='1
  • Rapid fire requests to check rate limiting (429)
BOLA (Broken Object Level Authorization) is the #1 API vulnerability. If you can access /api/users/456 while logged in as user 123, that's a critical bug.

Advanced Testing Techniques

These techniques separate good API testers from great ones.

Contract Testing

Verifies that the API matches its documented contract (usually OpenAPI/Swagger). Catches when the team changes an API response without updating the docs.

Tools
Pact, Dredd, Schemathesis
What it checks
Response schema, field types, required fields
If the API says it returns { "id": number } but returns { "id": string }, contract testing catches it.

Negative Testing Patterns

  • Send empty string where value is required
  • Send null for non-nullable fields
  • Exceed max field length (e.g. 10,000 char string in a "name" field)
  • Send wrong data types (string instead of integer)
  • Omit required fields entirely
  • Send extra unknown fields (mass assignment risk)
  • Non-existent resource IDs (e.g. GET /users/99999999)

Chaining Requests

Real-world API tests link multiple calls together. The output of one becomes the input of the next.

// Step 1: POST /auth/login → extract token // Step 2: POST /api/orders → extract order_id // Step 3: GET /api/orders/{order_id} → verify created // Step 4: DELETE /api/orders/{order_id} → cleanup
Always clean up test data. Use unique identifiers (timestamps or UUIDs) in test data to avoid conflicts with other tests.
